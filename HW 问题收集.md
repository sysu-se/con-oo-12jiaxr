## HW 问题收集

在 HW1 和 HW1.1 的开发过程中，我遇到了一些值得记录的问题。以下是其中 2~3 个已经解决的和 2~3 个尚未解决的主要挑战。

### 已解决

#### 1. 为什么修改对象内部字段后，界面不一定自动更新？

- **上下文**：  
  直接调用 `game.getSudoku().getGrid()[row][col] = value`，期望界面刷新，但没有任何变化。
- **解决手段**：  
  通过询问 AI ，了解到 Svelte 的响应式基于 **赋值操作**，而不是属性变更。直接修改对象内部字段不会改变对象的引用，因此 Svelte 无法检测到变化。  
  解决方案：所有修改都通过 `game.guess()` 方法完成，并在修改后手动调用 `gameStore.set(game)`，触发 store 的更新通知，进而驱动 UI 重新渲染。

#### 2. 深拷贝策略选择：`JSON.parse(JSON.stringify(...))` 与 `array.map(row => [...row])` 谁更好？

- **上下文**：  
  HW1 中使用了 `JSON.parse(JSON.stringify(...))` 来深拷贝 `grid` 和 `fixed` 数组，但老师上课时提示其性能较差。
- **解决手段**：  
  通过询问 AI 并对比两种方式的性能与适用场景，了解到在纯数字/布尔二维数组场景下，`grid.map(row => [...row])` 速度更快，且没有 JSON 序列化的副作用（如无法处理 `undefined`、函数等）。最终改用 `map + 展开运算符` 实现深拷贝。

### 未解决

#### 1. 笔记模式与历史栈的同步问题

- **上下文**：  
  用户填入数字后，系统会清除该格的所有笔记（候选数）。但是，当用户执行 **撤销（Undo）** 时，笔记应该恢复到之前的状态。当前设计中，笔记单独存储在 `candidates` store 中，没有被纳入历史快照，因此撤销时笔记不会自动回滚。
- **尝试解决手段**：  
  曾考虑将笔记也纳入 `Game` 的历史快照（即每次 `guess` 时同时保存笔记状态），但这会严重破坏领域对象的纯净性（笔记属于 UI 辅助状态，不应进入核心领域）。目前尚未找到既保持领域纯净又能正确同步笔记的方法。

#### 2. 响应式协议泄漏到组件事件处理里（需要手动 `gameStore.set`）

- **上下文**：  
  每次执行 `guess`、`undo`、`redo` 后，组件都必须手动调用 `gameStore.set(game)` 才能触发 UI 刷新。这种模式容易遗漏，且不符合“领域对象自行通知 View”的理想设计。
- **尝试解决手段**：  
  曾考虑让 `Game` 内部持有一个 Svelte store，在自身状态变更时自动 `set`，但这会污染领域对象，使其与 Svelte 框架耦合。目前仍保留手动 `set` 的方式，依靠代码审查和测试来保证完整性，尚未找到更优雅的自动通知方案。
